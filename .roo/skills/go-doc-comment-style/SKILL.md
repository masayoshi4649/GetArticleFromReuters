# GoDocコメント記述方式

Goの関数コメントを作成・修正する場合は、このスキルの形式に従う。

## 基本方針

- 関数直前に行コメントで記載する。
- ブロックコメント `/* ... */` は使用しない。
- コメントの先頭行は、GoDocとして認識されるように `関数名 は...` の形にする。
- `目的:` という見出しは使わず、先頭行で関数の概要を自然文として説明する。
- 先頭行のあとに空コメント行を入れ、`機能:`、`引数:`、`返り値:` の順で記載する。
- 箇条書きは `//   - ...` のインデント形式で記載する。
- 引数や返り値がない場合は `なし` と明記する。

## コメント構造

```go
// 関数名 は、関数の概要を説明します。
//
// 機能:
//   - 実行する処理を箇条書きで説明する
//   - 重要な分岐や副作用も必要に応じて説明する
//
// 引数:
//   - 引数名: 引数の意味を説明する
//   - なし
//
// 返り値:
//   - 型名: 返り値の意味を説明する
//   - error: エラー条件を説明する。成功時はnil
//   - なし
func example(arg string) (string, error) {
	return arg, nil
}
```

## 記述ルール

### 先頭行

- 必ず `関数名 は...` で始める。
- 関数が何をするものかを1文で説明する。
- 実装手順の詳細ではなく、呼び出し側から見た役割を説明する。

### 機能

- 関数内部で行う主要な処理を箇条書きで記載する。
- 3〜6項目程度を目安にする。
- ログ出力、DB更新、HTTP通信などの副作用がある場合は明記する。
- 異常系の扱いが重要な場合は明記する。

### 引数

- 各引数について `引数名: 説明` の形式で記載する。
- ポインタ、DBハンドル、HTTPクライアントなど、利用目的が重要な引数は用途も説明する。
- 引数がない場合は `なし` と記載する。

### 返り値

- 各返り値について `型名: 説明` の形式で記載する。
- 複数返り値の場合は返却順に記載する。
- `error` は、どのような場合に返るかと成功時に `nil` になることを説明する。
- 返り値がない場合は `なし` と記載する。

## 良い例

```go
// run は設定ファイル、PostgreSQL、Reuters APIを連携させ、未保存の記事をDBへ登録します。
//
// 機能:
//   - 設定ファイルを読み込む
//   - PostgreSQL接続を開く
//   - Reutersのセクション記事一覧を取得する
//   - 保存済み記事の詳細取得とDB保存をスキップする
//   - 未保存記事の詳細を取得してDBへ保存する
//   - 保存件数とスキップ件数を標準出力へ表示する
//
// 引数:
//   - なし
//
// 返り値:
//   - error: 処理に失敗した場合は原因を含むエラー。成功時はnil
func run() error {
	return nil
}
```

## 悪い例

```go
/*
run は処理を実行します。

## 引数

- ありません。

## 返り値

- errorを返します。
*/
func run() error {
	return nil
}
```

問題点:

- ブロックコメントを使用している。
- `機能:` がない。
- 引数と返り値の表現が統一されていない。
- `error` の発生条件と成功時の値が説明されていない。
